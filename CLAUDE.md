# AI 驱动学习平台

## 功能概述

UI 设计请参考: `.reference/index.html`
系统设计和开发计划参考: `docs`

### 知识库管理

1. 构建知识库：支持超链接、文本文档（txt, markdown 等）、word、PPT、excel、代码、图片等格式数据导入，内部进行结构化解析，生成图谱。解析方案可以参考：`.reference/graphify`：项目只需要集成其数据解析、整理、知识图谱构建、知识检索相关的功能
2. 知识库可视化：多数据源构建知识库，解析知识库图图谱进行可视化
3. 知识库检索：支持关键词检索知识库内容，子图查询等
4. 链接知识自动更新：针对以链接提供的知识，定时自动更新

### 基于知识库的内容生成

1. model provider: 支持 openai / anthropic / deepseek model provider
2. agent loop 核心逻辑：实现 agent loop 核心逻辑，支持 ai 工具调用
3. 检索工具：构建知识库图谱数据检索、查询工具
4. 知识库问答：基于 agent loop 实现知识检索、问答

## 开发规范

### 思考先行

**先想清楚再动手，不隐藏疑惑，不凭空假设。**

- 不确定的事情明确说出来，不猜、不编造
- 多种解读并存时，列出选项而非默认选一个
- 有更简单方案时直接指出，该拒绝就拒绝
- 遇到模糊点先停下来搞清楚，再继续

### 简单至上

**用最少的代码解决问题，不做投机性设计。**

- 不写需求之外的功能
- 不为单次调用造抽象
- 不预设"灵活性"或"可配置性"
- 不处理不可能发生的错误场景
- 200 行能写清楚的不写 500 行
- 三段相似代码好过一次不必要的抽象

### 手术式修改

**只动需要动的地方，只清理自己制造的垃圾。**

- 不改相邻代码的格式、注释、风格
- 不顺手重构不相干的东西
- 匹配已有代码风格，即使和你的偏好不同
- 你引入的 import / 变量 / 函数如果不再使用，删除干净
- 发现与当前任务无关的死代码，提出来但不擅自删除

### 目标驱动

**把模糊需求转化为可验证目标，循环直到确认通过。**

- "加个校验" → 先写无效入参的测试，让它通过
- "修这个 bug" → 先写复现测试，让它通过
- "重构 X" → 确保重构前后所有测试通过
- 多步任务先列步骤，每步说清验证标准

### 研发流程

- 遵循 `skill: karpathy-guidelines` 开发准则
- 方案设计、需求开发、bug 修复遵循 superpower 流程
- 编写代码采用 TDD 范式（先写测试，再写实现）
- UI 设计采用 `skill: frontend-design`
- 代码检索优先使用 `codegraph` 系列工具
- 编写后必须检视、评审、修改，直到满足规范
